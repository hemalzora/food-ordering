# Database Schema (per service)

Each service owns its own PostgreSQL database. These schemas were split out of the legacy
single MySQL database in `database/legacy/dumpsqlOSP.sql`.

## user_db — user-service

`users` — one row per account, with a role.

| Column | Type | Notes |
|---|---|---|
| id | BIGINT (identity) | primary key |
| name | VARCHAR(100) | |
| email | VARCHAR(120) | unique |
| password_hash | VARCHAR(100) | BCrypt hash |
| role | VARCHAR(20) | CUSTOMER, RESTAURANT or ADMIN (CHECK constraint) |
| phone / gender / address | VARCHAR | optional |
| active | BOOLEAN | |
| created_at | TIMESTAMP | |

From the legacy `admin`, `customer` and `restaurant` login tables, unified into one table with a `role`.

## restaurant_db — restaurant-service

- `restaurants` — id, name, email, phone, address, active.
- `menu_items` — id, restaurant_id (FK → restaurants), name, type, price `NUMERIC(9,2)`, description, active.

From the legacy `restaurant` and `menu` tables.

## order_db — order-service

- `menu_item_ref` — id, name, price, restaurant_id, active. A local copy of menu items the service uses to price a cart.
- `cart_items` — id, user_id, menu_item_id, quantity. Unique on (user_id, menu_item_id).
- `orders` — id, user_id, restaurant_id, status, total_amount, delivery_address, payment_method, created_at.
- `order_items` — id, order_id (FK → orders), menu_item_id, item_name, unit_price, quantity, line_total. Each line stores the name and price as a snapshot taken when the order is placed.

From the legacy `cart`, `orders` and `orderdetails` tables.

## Conventions

- One database per service. Cross-service references (`restaurant_id`, `user_id`, `menu_item_id`) are plain ids — there are no foreign keys across databases.
- Money is `NUMERIC(9,2)`; ids are `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- Each schema is created and seeded by Flyway (`V1__init.sql`) on the service's startup.
